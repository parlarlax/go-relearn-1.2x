# Q and A

## <https://github.com/golang-standards/project-layout> นี่ถือว่าเป็น bible ของชาว go รึเปล่า

## Answer

คำตอบสั้นๆ คือ **"ไม่ใช่ครับ"** และในความเป็นจริง มันเป็นประเด็นที่ถกเถียงกันอย่างมากในชุมชน Go ทั่วโลกครับ

แม้ว่าชื่อบัญชีผู้ใช้จะชื่อ `golang-standards` แต่โปรเจกต์นี้ **ไม่ใช่มาตรฐานอย่างเป็นทางการ (Official Standard)** จากทีมพัฒนาภาษา Go (Google) และมีชาว Go (Gophers) จำนวนมากที่ไม่เห็นด้วยกับการนำโครงสร้างนี้ไปใช้ในทุกโปรเจกต์

นี่คือประเด็นสำคัญที่คุณควรทราบครับ:

### 1. ทีมงาน Go เคยปฏิเสธสิ่งนี้อย่างเป็นทางการ

**Russ Cox** ซึ่งเป็น Lead ของโปรเจกต์ Go ที่ Google เคยไปตอบใน [Issue ของ Repository นี้](https://github.com/golang-standards/project-layout/issues/117) โดยระบุชัดเจนว่า: *"This is not a Go standard. ... This is just one person's opinion on how to organize a project."* (นี่ไม่ใช่มาตรฐานของ Go... มันเป็นเพียงความเห็นของคนคนหนึ่งในการจัดการโปรเจกต์เท่านั้น)

### 2. ทำไมมันถึงถูกวิจารณ์?

* **Over-engineering:** โครงสร้างนี้ซับซ้อนเกินไปสำหรับโปรเจกต์ขนาดเล็กถึงกลาง การมีโฟลเดอร์แยกย่อยจำนวนมาก (เช่น `/pkg`, `/internal`, `/api`, `/build`) ทำให้เกิดความยุ่งยากในการไล่ Code
* **ไม่ใช่สไตล์ของ Go:** ปรัชญาของ Go คือ **"Keep it simple"** และ **"Flat is better than nested"** แต่โปรเจกต์นี้ได้รับอิทธิพลมาจากภาษาอย่าง Java หรือ C# ซึ่งเน้นการจัดโครงสร้างแบบ Layered Architecture มากเกินไป
* **การใช้โฟลเดอร์ `/pkg`:** ในภาษา Go ยุคใหม่ (ตั้งแต่มี Go Modules) โฟลเดอร์ `/pkg` แทบจะไม่มีความจำเป็น และผู้เชี่ยวชาญหลายคนแนะนำว่าไม่ควรใช้หากไม่จำเป็นจริงๆ

### 3. แล้วโครงสร้างที่เป็น "มาตรฐานจริง" คืออะไร?

ในชุมชน Go เราจะเน้นสิ่งที่เรียกว่า **"Idiomatic Go"** ซึ่งมีแนวทางดังนี้ครับ:

* **เริ่มต้นให้เล็กที่สุด:** เอาไฟล์ `.go` ไว้ที่ Root ของโปรเจกต์เลยก็ได้ถ้าโปรเจกต์ยังเล็ก
* **`cmd/`:** ใช้สำหรับเก็บ Entry point (ไฟล์ `main.go`) แยกตามชื่อโปรแกรมที่ต้องการ Build
* **`internal/`:** เป็นโฟลเดอร์ "พิเศษ" ที่ Go Compiler จะอนุญาตให้เรียกใช้ Code ได้เฉพาะภายในโปรเจกต์เดียวกันเท่านั้น (นี่คือมาตรฐานที่ตัวภาษาบังคับใช้จริง)
* **เน้นความรับผิดชอบ (Functional):** แทนที่จะแบ่งเป็น `models/`, `controllers/`, `services/` (แบบ MVC) ให้แบ่งตามหน้าที่ เช่น `auth/`, `user/`, `payment/`

### สรุป: ควรใช้มันไหม?

* **ถ้าคุณทำ Microservices ขนาดใหญ่มาก** ที่มีคนทำงานร่วมกันเยอะๆ โครงสร้างนี้อาจจะช่วยให้การแบ่งส่วนงานดูชัดเจนขึ้น
* **ถ้าคุณเริ่มต้นโปรเจกต์ใหม่ หรือทำแอปทั่วไป** แนะนำให้เลี่ยงครับ เพราะจะทำให้โปรเจกต์ของคุณซับซ้อนเกินจำเป็น (Over-complicated)

**คำแนะนำ:** หากต้องการศึกษาโครงสร้างที่เป็นธรรมชาติของ Go จริงๆ แนะนำให้ดูโปรเจกต์ที่เป็น Standard Library ของ Go เอง หรือโปรเจกต์ Open Source ชื่อดังอย่าง **Docker** หรือ **Kubernetes** แทนครับ

## idiomatic คืออะไร

## Answer

คำว่า **Idiomatic** (อ่านว่า: อิด-ดี-อะ-แม-ติก) หากแปลตรงตัวจากรากศัพท์ภาษาศาสตร์จะหมายถึง **"การใช้ภาษาได้อย่างถูกต้องตามเจ้าของภาษา"** หรือ "เป็นไปตามสำนวนนิยม" ครับ

แต่ในโลกของการเขียนโปรแกรม **Idiomatic Code** หมายถึง **การเขียนโค้ดในรูปแบบที่ "ชุมชนนักพัฒนาภาษานั้นๆ" ยอมรับว่าดีที่สุด ถูกต้องที่สุด และสื่อสารได้เข้าใจง่ายที่สุด** ตามปรัชญาของภาษานั้นครับ

เพื่อให้เห็นภาพชัดขึ้น ผมขอสรุปประเด็นสำคัญดังนี้ครับ:

---

### 1. เปรียบเทียบกับภาษาคน (Analogy)

ลองนึกถึงภาษาไทยครับ:

* **โค้ดที่รันได้ (Valid):** "ขณะนี้ข้าพเจ้ามีความประสงค์จะรับประทานวารี" (เครื่องคอมพิวเตอร์เข้าใจว่าคุณจะกินน้ำ แต่มันฟังดูแปลกมาก)
* **โค้ดที่เป็น Idiomatic:** "หิวน้ำ" หรือ "กินน้ำหน่อย" (สั้น กระชับ เป็นสิ่งที่คนไทยใช้กันจริงๆ)

ในโปรแกรมมิ่งก็เช่นกัน คุณอาจจะเขียน Go ให้หน้าตาเหมือน Java (ใช้ Class, สืบทอดเยอะๆ) ซึ่งมัน **รันได้** แต่มัน **ไม่ใช่ Idiomatic Go** ครับ

### 2. ทำไมต้องเขียนแบบ Idiomatic?

* **Readability:** เพื่อให้คนอื่นในทีม (หรือตัวคุณเองในอีก 6 เดือนข้างหน้า) อ่านแล้วเข้าใจทันที เพราะมันเป็นรูปแบบที่ทุกคนคาดหวังจะเห็น
* **Maintainability:** โค้ดที่เขียนตามแนวทางของภาษา มักจะดูแลรักษาง่ายกว่าและมีบั๊กน้อยกว่า
* **Performance:** บ่อยครั้งที่แนวทาง Idiomatic ถูกออกแบบมาเพื่อให้ทำงานร่วมกับ Compiler หรือ Runtime ของภาษานั้นๆ ได้มีประสิทธิภาพสูงสุด

### 3. ตัวอย่าง Idiomatic Go vs. Non-Idiomatic

Go มีปรัชญาที่ชัดเจนมากคือ **"เรียบง่ายและชัดเจน" (Simple & Explicit)**

| เรื่อง | Non-Idiomatic (เขียนแบบภาษาอื่น) | Idiomatic Go (แบบที่ชาว Go ทำ) |
| :--- | :--- | :--- |
| **การจัดการ Error** | ใช้ `try-catch` (Go ไม่มี) หรือละเลย Error | **Check Error ทันที** ด้วย `if err != nil` |
| **ชื่อตัวแปร** | `indexLoopCounterVariable` (ยาวเกินไป) | `i` (สั้นๆ กระชับใน Scope เล็กๆ) |
| **การสร้าง Object** | สร้าง Class Hierarchy ซับซ้อน | ใช้ **Composition** (เอา Struct มาซ้อนกัน) |
| **การเช็คความยาว** | เขียนฟังก์ชัน `CheckIfListIsEmpty()` | ใช้ `if len(slice) == 0` |

---

### สรุปสั้นๆ

ถ้ามีคนบอกคุณว่า **"Your code is not idiomatic"** เขาไม่ได้บอกว่าโค้ดคุณผิดหรือรันไม่ได้นะครับ แต่เขาหมายความว่า **"โค้ดของคุณมันดูฝืนธรรมชาติของภาษานี้ไปหน่อย"**

> **สำนวนที่ฝรั่งชอบใช้:** "When in Rome, do as the Romans do"
> **ฉบับโปรแกรมเมอร์:** "When in Go, code as the Gophers do" ครับ

ในฐานะที่คุณเริ่มสนใจเรื่อง Project Layout ผมขอถามหน่อยครับว่า ตอนนี้คุณกำลังย้ายมาจากภาษาอื่น (เช่น Java หรือ Node.js) หรือเปล่าครับ? เผื่อผมจะช่วยเปรียบเทียบความแตกต่างให้เห็นภาพชัดขึ้น!
